How I built a CGPA calculator in a weekend — and what I learned

A walkthrough of the decisions behind the student GPA tool, from data structure choices to making it feel fast without a backend.

It started with a complaint from a coursemate. She had just finished her second semester exams and wanted to know her CGPA. She pulled out a calculator, a sheet of paper, and spent ten minutes doing arithmetic — only to realise she'd forgotten to account for a 2-unit course.

I watched this happen and thought: there has to be a better way. Not a complex app, not something that requires sign-up — just a simple, fast tool that does the math correctly and saves your data. That weekend, I sat down and built it.

The problem I was actually solving

Before writing a single line of code, I asked myself what the actual pain points were. Nigerian university students deal with a grading scale that not every online GPA calculator supports. The 5-point scale — where an A is 5.0, not 4.0 — is unique to most Nigerian institutions, and tools built for American universities are practically useless here.

Beyond that, students track GPA across multiple semesters, not just one. A tool that only computes a single semester's GPA misses the point entirely. What you need is a running CGPA across all your semesters combined.

The Nigerian 5-point grading scale: A = 5.0 · B = 4.0 · C = 3.0 · D = 2.0 · E = 1.0 · F = 0.0. Most online calculators assume a 4.0 scale — completely wrong for Nigerian students.

Choosing the data structure

The first real decision was how to represent the data. A CGPA calculation is fundamentally this formula:

// CGPA = Total Grade Points / Total Credit Units
// Grade Points for a course = Grade Point Value × Credit Units

const gradePoints = { A:5, B:4, C:3, D:2, E:1, F:0 };

function computeCGPA(semesters) {
  let totalPoints = 0, totalUnits = 0;
  semesters.forEach(sem => {
    sem.courses.forEach(course => {
      totalPoints += gradePoints[course.grade] * course.units;
      totalUnits  += course.units;
    });
  });
  return totalUnits > 0 ? totalPoints / totalUnits : 0;
}

The key insight: group courses by semester, not by year. This lets you compute both a per-semester GPA and a running CGPA from the same data, without duplicating any logic.

Making it fast without a backend

I didn't want users to create an account just to track their grades. That's friction that kills tools like this. The solution was localStorage — the browser's built-in key-value storage that persists across sessions.

// Save state on every change
function save() {
  localStorage.setItem('cgpa_data', JSON.stringify(state));
}

// Load state on page open
const state = JSON.parse(
  localStorage.getItem('cgpa_data') || 'null'
) || defaultState;

This means your data is there every time you open the tool — no server, no account, no cost. The trade-off is that data is tied to your browser, but for a student tool used on one device, that's fine.

UI decisions that mattered

Live recalculation

Every time you change a grade or credit unit, the CGPA updates instantly. No submit button, no loading. This feels obvious but requires discipline — every input needs an oninput handler that triggers a full recalculation and re-render.

Colour-coded GPA

The CGPA display changes colour based on your grade class — green for First Class, purple for Second Upper, amber for Second Lower, and red below that. This gives instant visual feedback without needing to read any text.

Semester naming

I let users rename their semesters. "100L 1st Semester" is the default, but some students use trimester systems or want custom labels. A simple editable input field on each semester tab solves this with zero extra complexity.

What I got wrong the first time

My first version had one big flaw: it recalculated and re-rendered the entire page on every keypress. While typing a course code like "MTH 101", the cursor would jump back to the start of the input after each character — because the input element was being destroyed and recreated on every render.

Lesson learned: When doing full DOM re-renders on input events, you lose focus and cursor position. The fix is to only re-render elements that don't have focus — or better, separate your data update from your render cycle.

The fix was simple: course code inputs update the state directly without triggering a full re-render. Only grade and unit dropdowns (which don't have cursor position issues) trigger the full recalculate-and-render cycle.

The result

The final tool supports unlimited semesters, per-semester GPA display, a running CGPA, colour-coded class of degree, and a text report you can download. It works offline, loads in under a second, and stores everything in your browser.

My coursemate who inspired it now uses it every semester. That's enough.
